## 2. 硕士论文写作指南（毕业论文）

### 2.1. 论文提纲

#### 2.1.1. 论文要点
本部分用于**对标盲审专家的评审逻辑**，在论文正式撰写前或撰写过程中，系统性检查论文设想是否达标。建议在论文初稿完成前至少完整填写一次，在送审前再次复查。

| 序号  | 要点   | 要点问题                         | 对应论文章节   | 回答及说明 | 备注          |
| --- | ---- | ---------------------------- | -------- | ----- | ----------- |
| 1   | 选题   | 要解决的问题是什么？问题是否明确、具体、可验证？     | 引言 / 第1章 |       |             |
| 2   | 研究意义 | 假设问题能解决，会带来什么理论或实践上的好处？      | 引言       |       |             |
| 3   | 研究现状 | 既然有这么大好处，以前有哪些研究尝试解决该问题？     | 相关工作     |       | 注意引用最新、权威文献 |
| 4   | 研究贡献 | 本研究相较已有工作，贡献或特色（不同且进步之处）在哪里？ | 引言 / 总结  |       |             |
| 5   | 研究方法 | 从理论和技术上，本研究采用了哪些方法支撑研究贡献？    | 方法章节     |       |             |
| 6   | 研究结果 | 有哪些实验或分析结果印证研究贡献？            | 实验章节     |       |             |
| 7   | 工作态度 | 工作量是否充足？研究过程是否体现系统性与完整性？     | 全文       |       | 非必填项        |


1. 工程项目需求与问题，输出：项目数据集；工程；取max: （wxy）；
2. DeepResearch查找参考文献和基线，输出: 可借鉴程序及其成熟度评分；工程；取max: （骨干，wxy）
3. 跑通基线（公开数据集+项目数据集）获得项目基线测试结果；取sum；
4. 观察项目问题域，头脑风暴提出改进点；取max：（本人，wxy）
5. 实现改进，测试排风险，验证改进有效；取sum；

- 选题：要解决的问题是什么？
	- 核心不是“你做了什么”，而是**你到底要解决什么问题**。
	  - 例如
		  - 不是光说“我做了一个系统 / 一个算法”
		  - 要明确成类似——**在保证任务完成的前提下，如何降低功耗**
		  - 这个问题本身是清晰的、具体的，而且是可以通过实验验证的。
	- 问题必须是**可验证的**，否则后面实验无法闭环；
	- 问题必须是**具体的**，不能是泛泛而谈；
	- 问题必须是**能写成一句话的**，否则引言第一段就写不下去。  
	- 如果你在写引言的时候发现“问题本身说不清楚”，那这篇论文基本就不用继续做了。

- 研究意义：假设问题解决，会带来什么好处？
	- 类比成项目申请书的“研究意义”。
	- 你不能只站在“我自己做了什么”的角度，而是要站在**整个研究领域**的角度来看
		- 如果这个问题被解决了，会带来什么价值？
	- 典型角度包括：
		- 理论层面：是否补充或完善了某类方法、模型或理论；
		- 实践层面：是否解决了工程中的真实痛点，比如续航、稳定性、可部署性等。
	- 研究意义不是你自己觉得有用，而是要让“做这个方向的人”（审稿）都能理解为什么有用。
	- 这一步直接决定了引言能不能“站得住”。

- 研究现状：以前有哪些研究尝试解决这个问题？
	- 既然你说这个问题这么重要，那**别人一定尝试过解决**，你必须搞清楚他们是怎么做的。
	- 不查文献就做科研，本质上是“蒙着眼睛走路”；
	- 查文献不是为了堆引用，而是为了回答“为什么还轮得到你来做”。
		1. **必须查到最新**，否则存在“别人已经做完你还不知道”的风险；
		2. 即使你的问题来自工程项目，也不能跳过这一步。

- 研究贡献：你的不同和进步在哪里？
	- 逻辑：
		- 前面已经说明了问题的重要性；
		- 也说明了别人已经做了什么；
		- 那么自然就要回答：**你到底多做了什么？**
	- 贡献点不是最后硬凑的，而是**从前面的逻辑自然推导出来的**；
		- 一般是 2–3 条，写在引言结尾；
		- 如果你在写引言时发现贡献点“写不出来”，
	**大多数论文被拒，不是实验不行，而是贡献点在引言里就站不住。**

- 研究方法：你是如何支撑这些贡献的？
	- 关键在于研究流程是否合理
		- **不要一上来就写完整程序、跑大规模实验，而是先做预研、排风险**
	- 先验证原理是否成立；
	- 先区分问题是 Bug 还是方法本身的问题；
	- 哪怕程序很“脏”、不稳定，只要能验证原理就够了
	- 方法章节本质上要回答的是：  
		- 你采用了哪些理论和技术手段，使得你的贡献不是“拍脑袋”，而是有技术支撑的。

- 研究结果：实验如何印证研究贡献？
	- 实验不仅仅是为了展示工作量，而是为了证明贡献点。
	- 方法 → 实验 → 贡献
		- 实验设计是否公平（基线是谁、条件是否一致）；
		- 实验结果是否能直接支撑你在引言中 claim 的创新点；
		- 结果出来后，能否清楚地解释“为什么是这样”。
	- 实验跑得多不等于实验做得好，**关键是是否“对着问题跑”**。

-  工作态度：是否体现系统性与完整性？
	- 硕士阶段看的不是“你是不是天才”，而是你是否具备**基本的科研素养**；
		- 是否有完整流程；
		- 是否能反复验证、修正；
		- 是否能把逻辑讲清楚并自洽。
	- **评审在看的是你能不能作为一个“合格的研究生”独立完成一项研究。**

### 2.2. 论文评审重点部分

从盲审专家角度，以下内容通常是**优先阅读和重点关注**的部分：

- 摘要
- 研究现状与研究意义
- 参考文献质量与规范性
- 研究贡献点是否清晰
- 研究内容与关键问题定义
- 实验结果（图表与分析）

### 2.3. 论文常见问题

#### 2.3.1. 论文语言文字问题

语言文字问题是评审专家最先感知到的问题，往往会直接影响对论文整体态度的判断。常见问题包括：

- 错别字
- 病句
- 表述前后不一致、前言不搭后语

常用修改方法包括：

- 向他人朗读全文，检查错别字与病句
- 缩句，验证语句是否通顺
- 扩句，将关键技术细节明确写清
- 梳理逻辑顺序，合理使用“因为……所以……”“虽然……但是……”等逻辑连接

#### 2.3.2. 逻辑问题

逻辑问题主要分为两类：

- **逻辑矛盾**：前后表述相互冲突
- **逻辑不完备**：提出的问题或目标，在后文中未被完整回应

例如：  
在需求分析中指出系统难点为“吞吐率、低冗余、可扩展性”，但实验部分却未对吞吐率、冗余性或扩展性进行任何验证。

#### 2.3.3. 篇幅不够的问题

推荐使用“**细化思路，看图写话**”的方法扩展篇幅，而非简单重复文字。

例如：  
提出“通过发送心跳包测量网络延迟”的思路时，可进一步补充：

- 时序图说明测试原理
- 流程图说明实现流程
- 参数设置与实验环境说明

这种扩展方式通过引入新信息（图示、流程、分析），不会造成论文冗余，反而提升表达清晰度。

#### 2.3.4. 文不对题的问题

常见表现为：

- 题目中声明的关键技术、特色或创新点，在正文中未得到体现
- 论文内容与题目关注点偏离

建议在定稿前，逐条对照题目关键词，检查正文是否有明确对应内容。

#### 2.3.5. 实验系统无法运行的问题

实验系统可以不完善、不稳定，但**必须能够基本运行并产生实验数据**。  
评审专家更关注：

- 是否验证了论文提出的核心思想
- 实验是否能支撑论文结论

而非系统是否达到工程级稳定性。
