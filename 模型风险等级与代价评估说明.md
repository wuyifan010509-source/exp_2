# 模型风险等级与代价评估模型详解

## 1. SLM 风险等级预测

### 1.1 模型架构
- **基础模型**: Qwen/Qwen2.5-0.5B（0.5B参数）
- **微调方式**: LoRA (r=16, alpha=32)
- **任务类型**: 三分类（Low/Mid/High）
- **目标层**: q_proj, v_proj

### 1.2 训练数据映射
```python
COST_LABEL_MAP = {
    0: 1,      # Low  - 错误代价 1元
    1: 50,     # Mid  - 错误代价 50元  
    2: 1000,   # High - 错误代价 1000元
}
```

### 1.3 核心创新：非对称损失函数
**文件位置**: `exp/slm_distillation/asymmetric_loss.py:64-134`

**惩罚系数**:
```python
ALPHA_UNDERPREDICT = 12.0   # 漏报惩罚（把高危预测为低危）
ALPHA_OVERPREDICT = 1.0     # 误报惩罚（把低危预测为高危）
```

**代价矩阵**（行=真实类别, 列=预测类别）:

| 真实\预测 | Low | Mid | High |
|-----------|-----|-----|------|
| Low       | 0   | 1   | 1    |
| Mid       | 12  | 0   | 1    |
| High      | 12  | 12  | 0    |

**损失函数公式**:
```
L = α * max(0, y_i - ŷ_i)² + max(0, ŷ_i - y_i)²
```

其中:
- y_i: 真实标签
- ŷ_i: 模型预测
- α: 漏报惩罚系数（12.0）

## 2. 错误代价评估模型

### 2.1 代价配置
**文件位置**: `exp/scheduling_simulation/config.py:12-24`

```python
COST_CONFIG = {
    "error_cost": {
        "low": 1,      # 低风险答错损失
        "mid": 10,     # 中风险答错损失
        "high": 1000,  # 高风险答错损失
    },
    "latency_cost_per_ms": 0.0003,  # 延迟代价：0.3元/秒
    "human_fixed_cost": 50,         # 人工固定成本
}
```

### 2.2 期望损失计算公式

#### LLM期望损失
```
Cost_LLM = P(error) × C_error + Latency_LLM × C_latency
         = llm_error_prob × true_cost + 16ms × 0.0003
```

其中:
- `llm_error_prob`: 模型预测的LLM错误概率
- `true_cost`: 真实风险等级对应的错误代价
- `16ms`: SLM推理延迟

#### 人类期望损失（实时计算）
```
Cost_Human = C_human_fixed + Expected_Wait × C_latency
           = 50 + queue_wait_seconds × 1000 × 0.0003
```

其中:
- `queue_wait_seconds`: 当前队列预期等待时间（秒）
- `Expected_Wait` 通过 M/M/c 排队模型实时计算

### 2.3 路由决策规则
**文件位置**: `exp/scheduling_simulation/routers.py:217-271`

```python
if Cost_LLM > Cost_Human:
    assign_to = "human"  # 转人工处理
else:
    assign_to = "llm"    # 给LLM处理
```

**决策逻辑说明**:
1. **LLM-Only**: 所有请求全部给LLM，无人类介入
2. **Vanilla L2D**: P(LLM错) > 阈值(0.2) 时转人工
3. **Static Cost**: 静态比较，LLM损失 > 100元时转人工
4. **Dynamic (Ours)**: 实时感知队列状态，动态决策

## 3. 评估指标与可视化

### 3.1 核心评估指标

| 指标 | 说明 |
|------|------|
| **ETC** | Expected Total Cost（总期望成本） |
| **CER** | Catastrophic Error Rate（灾难性错误率） |
| **SLA** | 高价值请求最大等待时间约束（30秒） |
| **CSA** | Cost-Sensitive Accuracy（代价敏感准确率） |

### 3.2 CER（灾难性错误率）计算规则
**文件位置**: `simulator.py:616-618`

```python
# 所有High风险（无论P(error)多大）都计入
if r.true_risk_level == "high":
    catastrophic_errors += r.llm_error_prob
```

**计入条件**:
1. 分配给LLM的请求
2. **High风险等级**（无论P(error)多大，全部计入）

**不计入的情况**:
- 转人工处理的High风险请求
- Mid/Low风险请求（无论P(error)多大）

QUEUE_CONFIG = {

## 4. 实验结果对比

### 4.1 四种策略性能对比

| 指标 | LLM-Only | Vanilla L2D | Static Cost | Dynamic (Ours) |
|------|----------|-------------|-------------|----------------|
| **ETC (总期望成本)** | ~44,000 | ~102,000 | ~43,000 | **~29,000** |
| **CER (灾难性错误率)** | 0.1360 | 0.0000 | 0.0000 | **0.0496** |
| **平均排队延迟(ms)** | 0 | ~900,000 | ~700,000 | **~300,000** |
| **人工转交率** | 0% | 36% | 18% | **11%** |
| **High风险转交率** | 0% | 33% | **100%** | 63% |

### 4.2 关键结论

1. **Dynamic Queue-Aware策略**通过实时感知队列状态，在保证高危问题大部分转人工的同时，显著降低总体成本和排队延迟

2. **成本分解**:
   - LLM-Only: 主要由Error Cost构成（~44,000元）
   - Vanilla L2D: 主要由Latency Cost构成（~96,000元排队延迟代价）
   - Static Cost: 平衡Error和Latency
   - Dynamic: 优化后的综合成本最低（~29,000元）

3. **风险等级处理**:
   - Static Cost策略对High风险100%转人工，确保零灾难性错误
   - Dynamic策略在成本和风险之间取得平衡，CER仅为0.0496
