# 人机混合调度策略对比文档

## 概述

本文档总结 `exp-2/scripts/run_priority_comparison.py` 中实现的四种人机转接调度策略，详细说明各策略的核心逻辑、决策依据和性能对比。

---

## 策略一：Vanilla L2D (传统阈值路由)

### 核心逻辑
仅基于模型信心度（错误概率）做决策，不考虑实际损失代价和排队延迟。

```python
if P(LLM错) > threshold:  # threshold = 0.5
    转人工
else:
    LLM处理
```

### 决策依据
- **单一指标**：模型错误概率 P(error)
- **固定阈值**：0.5
- **无代价意识**：不区分不同风险等级的实际业务损失

### 特点
| 维度 | 说明 |
|------|------|
| 队列感知 | ❌ 无，不考虑排队延迟 |
| 优先级区分 | ❌ 无，所有请求同等对待 |
| 人工排队延迟 | 57.0秒 |
| 转人工率 | 35.3% |

### 局限性
- High风险问题即使错误概率为0.49（接近阈值）也会走LLM
- 不考虑人工队列拥堵情况
- 无法处理极端高风险场景（如自杀倾向）

---

## 策略二：Static Cost (静态代价路由)

### 核心逻辑
引入代价概念，比较LLM期望损失与固定人类成本阈值。

```python
R_LLM = P(LLM错) × C_error        # LLM期望损失
if R_LLM > C_human_threshold:     # C_human = 50元
    转人工
else:
    LLM处理

# 人类期望损失（静态估计）
cost_human = 100元 + 30秒 × 延迟单价  # 假设固定等待30秒
```

### 决策依据
- **代价敏感**：考虑错误代价 C_error（High=2000元, Mid=10元, Low=1元）
- **静态估计**：假设人工处理平均等待30秒（固定值）
- **无视队列状态**：不感知实时排队情况

### 特点
| 维度 | 说明 |
|------|------|
| 队列感知 | ❌ 无，固定假设30秒等待 |
| 优先级区分 | ❌ 无 |
| 人工排队延迟 | 94.7秒（最长） |
| 转人工率 | 35.3% |

### 局限性
- 静态估计无法反映实时队列拥堵
- 在高峰期会低估实际等待时间，导致过多转人工
- 排队延迟最长（94.7秒），用户体验最差

---

## 策略三：Dynamic Queue-Aware (动态队列感知) ⭐ Ours

### 核心逻辑
实时比较LLM与人类的期望损失，动态感知队列状态做决策。

```python
# 1. LLM期望损失（实时）
cost_llm = P(LLM错) × C_error + LLM_latency_cost

# 2. 人类期望损失（实时队列状态）
queue_state = 获取当前队列状态()
expected_wait = queue_state["expected_wait"]  # 实时预期等待时间(秒)
cost_human = 100元 + expected_wait × 延迟单价

# 3. 动态决策
if cost_llm > cost_human:
    转人工
else:
    LLM处理
```

### 决策依据
- **实时队列感知**：获取当前队列长度、预期等待时间
- **动态代价比较**：LLM损失 vs 人类损失（含延迟成本）
- **自适应调节**：队列拥堵时少转人工，队列空闲时多转人工

### 特点
| 维度 | 说明 |
|------|------|
| 队列感知 | ✅ 实时获取预期等待时间 |
| 优先级区分 | ❌ 所有请求共享统一队列 |
| 人工排队延迟 | 45.0秒 |
| 转人工率 | 35.3% |

### 优势
- 能根据实时队列状态自适应调节转人工策略
- 避免高峰期过度转人工导致队列拥堵
- 相比Static Cost减少52%排队延迟

### 局限性
- 所有风险等级共享同一队列，High风险可能被Mid/Low阻塞
- High风险没有优先处理权

---

## 策略四：Dynamic Priority Queue (优先级队列) ⭐ Ours+

### 核心逻辑
在Dynamic基础上增加**非抢占式优先级队列**，High风险任务优先处理。

```python
# 三路独立优先级队列
High队列: H → H → H...  (新来的H排在现有H后面)
Mid队列:  M → M → M...
Low队列:  L → L → L...

# 按风险等级分别计算预期等待时间
expected_wait = {
    "high": High队列预期等待,
    "mid":  Mid队列预期等待,
    "low":  Low队列预期等待
}[risk_level]

# 分别计算人类期望损失
cost_human = 100元 + expected_wait × 延迟单价

# 决策
if cost_llm > cost_human[risk_level]:
    转对应优先级队列
```

### 优先级队列调度规则
1. **优先级**: High > Mid > Low
2. **非抢占式**: 正在处理的任务不中断
3. **服务器空闲时按优先级取任务**
4. **同优先级FIFO**: 先来的High优先于后来的High

### 决策依据
- **实时队列感知**：各优先级队列独立计算预期等待
- **优先级保障**：High风险使用更短的预期等待时间
- **成本比较**：与Dynamic相同，但等待时间更优

### 特点
| 维度 | 说明 |
|------|------|
| 队列感知 | ✅ 三队列独立计算预期等待 |
| 优先级区分 | ✅ High/Mid/Low三路队列 |
| 人工排队延迟 | **30.5秒**（最短） |
| 转人工率 | 35.3% |
| High风险转接率 | 100% |

### 优势
- **High风险100%转人工**：所有12个High风险OOS问题都转人工
- **排队延迟最短**：30.5秒，比Dynamic减少32%
- **零灾难性错误**：High风险全部由人工处理
- **整体平均时延最优**：1.76秒

### 核心不等式
```
P(LLM错) × C_error  vs  E[该优先级等待时间] × C_delay + C_human_fixed
```

## 策略对比总结

| 策略 | 决策核心 | 队列感知 | 优先级区分 | 人工排队延迟 | 整体平均时延 | High转接率 |
|------|---------|---------|-----------|------------|------------|-----------|
| **LLM-Only** | 全走LLM | ❌ | ❌ | 0秒 | 35.9ms | 0% |
| **Vanilla L2D** | P(error) > 0.5 | ❌ | ❌ | 57.0秒 | 3.26秒 | 100% |
| **Static Cost** | 期望损失 > 50元 | ❌ 固定30秒 | ❌ | 94.7秒 | 5.39秒 | - |
| **Dynamic** | 实时成本比较 | ✅ 统一队列 | ❌ | 45.0秒 | 2.58秒 | 100% |
| **Dynamic Priority** | 实时成本 + 优先级 | ✅ 三队列 | ✅ | **30.5秒** | **1.76秒** | **100%** |

---

## 关键发现

### 1. 转人工率一致性
所有有人工调度的策略转人工率均为 **35.3%**（12/34），说明：
- 在相同错误概率分布下，各策略识别的"需要人工"问题数量一致
- 差异主要体现在**排队延迟**和**High风险覆盖率**

### 2. 排队延迟优化效果

| 策略 | 排队延迟 | vs Dynamic 改善 |
|------|---------|----------------|
| Static Cost | 94.7秒 | -110% |
| Vanilla L2D | 57.0秒 | -27% |
| Dynamic | 45.0秒 | 基准 |
| **Dynamic Priority** | **30.5秒** | **+32%** |

### 3. High风险处理效果

| 策略 | High风险转接率 | 灾难性错误率 |
|------|--------------|------------|
| LLM-Only | 0% | 26.5% |
| Vanilla L2D | 100% | 0% |
| Dynamic | 100% | 0% |
| **Dynamic Priority** | **100%** | **0%** |

### 4. 时延与安全的权衡

**纯LLM方案**:
- ✅ 平均时延最低（35.9ms）
- ❌ 灾难性错误率 26.5%（High风险走LLM）

**Dynamic Priority方案**:
- ✅ High风险100%转人工，零灾难性错误
- ✅ 整体平均时延仅 1.76秒（可接受）
- ✅ 排队延迟最短（30.5秒）

---

## 结论

**Dynamic Priority Queue (Ours+)** 是最优策略：

1. **安全性**：High风险OOS问题 100% 转人工，零灾难性错误
2. **效率性**：平均排队延迟 30.5秒，整体平均时延 1.76秒
3. **公平性**：优先级队列确保高风险问题优先处理
4. **自适应性**：根据实时队列状态动态调节转人工决策

相比纯LLM方案，增加 1.73秒平均时延，换来：
- High风险全部人工处理
- 避免潜在的重大业务损失（如自杀危机、资金损失）
