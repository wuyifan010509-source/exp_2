**非对称损失函数设置**（从 `exp/slm_distillation/config.py` 和 `asymmetric_loss.py`）：

**核心公式**（分类任务）：
```python
# 代价矩阵 [真实类别, 预测类别]
cost_matrix[i, j] = α   # 当 j < i (漏报)
cost_matrix[i, j] = 1.0 # 当 j > i (误报)
cost_matrix[i, i] = 0.0 # 正确分类
```

**具体权重**:

| 错误类型 | 惩罚权重 | 说明 |
|:--------:|:--------:|:----:|
| **漏报** (预测 < 真实) | **α = 12.0** | 将High预测为Low/Mid |
| **误报** (预测 > 真实) | **1.0** | 将Low/Mid预测为High |
| **正确分类** | 0.0 | 无惩罚 |

**映射到风险等级**:
- Low (0) → Mid/High: 权重 1.0
- Mid (1) → High: 权重 1.0
- Mid (1) → Low: 权重 12.0
- **High (2) → Mid/Low: 权重 12.0** ← 最严格

**基线模型参数设定**：

**BGE-small-zh（语义路由）**:
- **模型**: BAAI/bge-small-zh-v1.5
- **向量维度**: 512（模型默认）

## 📊 6.2.2 核心实验：风险评估器性能对比

### ✅ 可填写内容

#### 1. 评价指标定义

**准确率 (Accuracy)**:
- 计算方式: 正确预测数 / 总样本数
- SLM结果: 74.75%

**High-Recall (高危召回率)**:
- 计算方式: 正确识别的高危数 / 实际高危数
- **SLM结果: 96.97%** ← 核心亮点
- **重要性**: 金融场景"宁可误报一千，不可漏报一个"

**总期望代价 (Total Expected Cost)**:
- 公式: `Error Cost + Latency Cost + Human Cost`
- SLM结果: **314.01**（最优）

**推理延迟 (Latency)**:
- TTFT（首字延迟）或端到端耗时
- **SLM平均: 15.74 ms**
- **BGE语义路由: 1.0 ms**
- **32B大模型: 94.90 ms**

#### 2. 图表展示

**核心指标对比表**（99条测试样本）：

| 指标 | 语义路由 | **SLM 蒸馏** | 32B 大模型 |
|------|:--------:|:------------:|:----------:|
| **准确率** | 65.66% | **74.75%** | 64.65% |
| **高危Recall** | 75.76% | **96.97%** | 54.55% |
| **漏报数** | 12 | **1** | 15 |
| **误报数** | 22 | 24 | **20** |
| **平均延迟** | **1.0 ms** | 15.74 ms | 94.90 ms |
| **★ 总期望代价** | 921.05 | **314.01** | 1489.65 |

**各类别Recall详细对比**:

| 方法 | Low Recall | Mid Recall | **High Recall** |
|------|:----------:|:----------:|:---------------:|
| 语义路由 | 48.48% | 72.73% | 75.76% |
| SLM蒸馏 | 42.42% | **84.85%** | **96.97%** |
| 32B大模型 | 42.42% | **96.97%** | 54.55% |

**混淆矩阵对比**（建议绘制图表）：

**关键观察**: 32B模型将15个High样本错误预测为Mid（漏报严重），而SLM仅漏报1个。

#### 3. 结果深度分析要点

**BGE拉胯的原因**（代码层面）：
- 仅基于字面相似度（余弦相似度），无法理解金融概念深度
- 无训练过程，无法学习风险模式
- 依赖锚点质量，锚点有限（仅45条）

**32B High-Recall低的原因**（从实验观察）：
- 通用大模型倾向于"盲目自信"
- Prompt设计简单，缺乏金融风险领域知识注入
- 未经过非对称损失"保守性训练"

**SLM赢的原因**（代码+实验验证）：
- 参数仅0.5B，但LoRA微调专注特定任务
- 非对称损失（α=12）强制模型保守
- 15.74ms延迟满足网关前置拦截要求

## 🔀 6.2.3 分配策略对比 (端到端仿真)

### ✅ 可填写内容

#### 1. 基线策略的具体运行机制

**策略1：LLM-Only**（从 `routers.py:50-80`）
- **机制**: 所有请求全由LLM回答
- **期望损失**: `P(error) × C_error`
- **死穴**: 无人工介入，高风险问题可能答错

**策略2：Vanilla L2D (置信度阈值)**（从 `routers.py:83-131`）
- **阈值**: `P(LLM错) > 0.2` 时转人工
- **机制**: 仅看模型信心，不考虑排队情况
- **死穴**: 可能把人工队列撑爆（仿真结果：人工处理109条，排队延迟883秒）

**策略3：静态代价敏感路由**（从 `routers.py:134-185`）
- **阈值**: `R_LLM > 100元` 时转人工
- **机制**: 计算风险代价，但等待时间设为静态常数30秒
- **死穴**: "只认死理"，无视当前队列拥堵情况

**策略4：本文动态调度 (Dynamic Queue-Aware)**（从 `routers.py:188-270`）
- **核心不等式**:
  ```
  if P(LLM错) × C_error > E[实时排队延迟] × C_delay + C_human:
      转人工
  else:
      LLM处理
  ```
- **机制**: 实时获取队列状态，动态计算期望等待时间

#### 2. 指标定义

**CER (灾难性错误率)**:
- 高风险问题被LLM错误回答的占比
- 计算: `catastrophic_errors / total_requests`
- **本文结果**: 0.0496 (4.96%)

**平均排队延迟**:
- 用户真实等待时间（毫秒）
- **本文结果**: 308,516 ms (约5分钟)
- **对比**: Vanilla L2D高达883,482 ms (约15分钟)

**总期望成本 (ETC)**:
- 计算: `错误代价 + 延迟代价 + 人工成本`
- **本文结果**: 29,789.12元（最优）
- **LLM-Only**: 44,370元
- **Vanilla L2D**: 102,405元
- **Static Cost**: 42,760元

#### 3. 结果深度分析（基于仿真数据）

**各策略ETC对比**（300条请求）：

| 策略 | ETC | 错误代价 | 延迟代价 | 人工成本 |
|:----:|:---:|:--------:|:--------:|:--------:|
| LLM-Only | 44,370 | 44,365 | 5 | 0 |
| Vanilla L2D | 102,405 | 5 | 96,300 | 6,100 |
| Static Cost | 42,760 | 3,565 | 37,845 | 1,350 |
| **Dynamic (Ours)** | **29,789** | 18,445 | 10,493 | 850 |

**关键发现**:
- **Dynamic策略节省成本**: 比LLM-Only省33%，比Vanilla省71%
- **人工转交率**: 仅11.33%（LLM处理266条，人工34条）
- **智能分流**: 高峰期将Low/Mid风险放行给LLM，保留人工带宽给High风险

**队列状态决策示例**（从仿真日志推断）:
```python
# 当队列空闲时
if queue_length=0, busy_servers=2:
    expected_wait = 0s
    cost_human = 50元  # 几乎无延迟代价
    → 高风险请求转人工

# 当队列拥堵时  
if queue_length=50, busy_servers=5:
    expected_wait = 600s
    cost_human = 50 + 600×0.0005×1000 = 350元  # 延迟代价300元
    → 中低风险请求可能直接给LLM
```
