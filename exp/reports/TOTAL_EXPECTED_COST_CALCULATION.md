# 总期望代价计算方法详解

> 本文档详细解释风险评估器实验中 Total Expected Cost 的计算方式，包含两种评估场景的具体公式和示例。

---

## 📊 概述

实验中涉及**两种评估场景**，总期望代价的计算方式略有不同：

1. **风险评估器性能评估** (`evaluate.py`): 对比语义路由 vs SLM vs 32B大模型的分类性能
2. **调度策略对比** (`simulator.py`): 对比四种路由策略的端到端性能

---

## 场景一：风险评估器性能评估

**代码位置**: `exp/evaluation/evaluate.py`

**适用场景**: 对比三种风险分类方法（语义路由、SLM蒸馏、32B大模型）

### 核心参数配置

```python
# 代码位置: evaluate.py:34-51
COST_MAP = {
    "low": 1,      # 低风险错误代价
    "mid": 50,     # 中风险错误代价  
    "high": 1000,  # 高风险错误代价
}

C_HUMAN = 200  # 人工转接成本

ERROR_COST_MULTIPLIER = {
    "underpredict": 10.0,  # 漏报系数（预测<实际）
    "overpredict": 1.0,    # 误报系数（预测>实际）
}

LATENCY_COST_COEFF = 0.01  # 延迟代价系数（元/ms）
```

### 1. Error Cost（错误代价）

**计算公式**:
```python
def compute_error_cost(true_level, pred_level):
    if pred == true: 
        return 0.0
    
    cost_diff = abs(COST_MAP[true] - COST_MAP[pred])
    
    if pred < true:  # 漏报：高危被预测为低危
        return cost_diff × 10.0 + 200
    else:  # 误报：低危被预测为高危
        return cost_diff × 1.0
```

**计算示例**:

| 真实等级 | 预测等级 | 错误类型 | 计算过程 | 代价 |
|:--------:|:--------:|:--------:|:---------|:----:|
| **High** | **Low** | 漏报 ❌ | \|1000-1\|×10 + 200 | **9,990元** |
| **Mid** | **Low** | 漏报 ❌ | \|50-1\|×10 + 200 | **690元** |
| **Low** | **High** | 误报 ⚠️ | \|1-1000\|×1 | **999元** |
| **Mid** | **High** | 误报 ⚠️ | \|50-1000\|×1 | **950元** |
| High | High | 正确 ✅ | 0 | **0元** |
| Mid | Mid | 正确 ✅ | 0 | **0元** |
| Low | Low | 正确 ✅ | 0 | **0元** |

**设计意图**: 漏报惩罚是误报的10倍，体现金融场景"宁可误报，绝不漏报高危"的保守性。

### 2. Latency Cost（延迟代价）

**计算公式**:
```python
latency_cost = latency_ms × 0.01
```

**计算示例**:

| 方法 | 延迟 | 计算 | 延迟代价 |
|:----:|:----:|:----:|:--------:|
| 语义路由 | 1.0 ms | 1.0 × 0.01 | **0.01元** |
| SLM蒸馏 | 15.74 ms | 15.74 × 0.01 | **0.1574元** |
| 32B大模型 | 94.9 ms | 94.9 × 0.01 | **0.949元** |

### 3. Total Expected Cost（总期望代价）

**计算公式**:
```python
# 平均到每条样本
avg_error_cost = total_error_cost / n
avg_latency_cost = (total_latency × 0.01) / n
total_expected_cost = avg_error_cost + avg_latency_cost
```

**注意**: 此场景不考虑 Human Cost，因为是纯分类器对比，不涉及调度决策。

### 实际结果示例（99条测试样本）

| 方法 | 总错误代价 | 平均错误代价 | 延迟代价 | **总期望代价** |
|:----:|:----------:|:------------:|:--------:|:------------:|
| 语义路由 | 91,182.96 | 921.04 | 0.01 | **921.05** |
| **SLM蒸馏** | 31,071.15 | **313.85** | 0.16 | **314.01** ✅ |
| 32B大模型 | 147,381.30 | 1,488.70 | 0.95 | **1,489.65** |

## 场景二：调度策略对比

**代码位置**: `exp/scheduling_simulation/simulator.py`

**适用场景**: 对比四种路由策略（LLM-Only、Vanilla L2D、Static Cost、Dynamic Queue-Aware）

### 核心参数配置

```python
# 代码位置: config.py:11-24
COST_CONFIG = {
    "error_cost": {
        "low": 1,      # 元
        "mid": 10,     # 元
        "high": 1000,  # 元
    },
    "latency_cost_per_ms": 0.0005,  # 元/ms
    "human_fixed_cost": 50,  # 元/次
}

LLM_LATENCY = {
    "slm": 16,      # ms
    "llm_32b": 95,  # ms
}
```

### 1. Error Cost（期望错误代价）

**针对LLM处理**:
```python
if r.assigned_to == "llm":
    # 期望错误代价 = P(错) × 真实错误代价
    expected_error_cost = r.llm_error_prob × r.true_cost
```

**计算示例**:

| 风险等级 | LLM错误概率 | 真实代价 | 期望错误代价 |
|:--------:|:-----------:|:--------:|:------------:|
| **High** | 0.80 | 1000元 | 0.8 × 1000 = **800元** |
| **Mid** | 0.40 | 10元 | 0.4 × 10 = **4元** |
| **Low** | 0.10 | 1元 | 0.1 × 1 = **0.1元** |

**注意**: 人工处理无错误代价（假设人类不会答错）。

### 2. Latency Cost（延迟代价）

**针对LLM处理**:
```python
latency_cost = LLM_LATENCY["slm"] × 0.0005  # 16 × 0.0005 = 0.008元
```

**针对人工处理**:
```python
wait_cost = r.waiting_time × 1000 × 0.0005  # 秒→毫秒×系数
```

**计算示例**:

| 处理方式 | 延迟 | 计算 | 延迟代价 |
|:--------:|:----:|:----:|:--------:|
| LLM推理 | 16 ms | 16 × 0.0005 | **0.008元** |
| 人工等待60秒 | 60,000 ms | 60000 × 0.0005 | **30元** |
| 人工等待300秒 | 300,000 ms | 300000 × 0.0005 | **150元** |

### 3. Human Cost（人工成本）

**计算公式**:
```python
total_human_cost += 50  # 每次转人工固定50元
```

**说明**: 模拟人工客服的人力成本，每次转接收取固定费用。

### 4. Total Expected Cost（总期望代价）

**计算公式**:
```python
expected_total_cost = total_error_cost + total_latency_cost + total_human_cost
```

**注意**: 这是300条请求的**总代价**，非平均代价。

### 案例：Dynamic Queue-Aware 策略（300条请求）

**数据统计**:
- 总请求: 300条
- LLM处理: 266条
- 人工处理: 34条

**代价分解**:

| 代价类型 | 计算过程 | 数值 |
|:--------:|:---------|:-----|
| **Error Cost** | Σ(P(error) × C_error) 对266条LLM请求 | **18,445.31元** |
| **Latency Cost** | LLM: 266×0.008 + 人工: 34×等待代价 | **10,493.81元** |

## 🔑 关键设计思想

### 1. 非对称惩罚设计

| 错误类型 | 惩罚系数 | 设计意图 |
|:--------:|:--------:|:--------:|
| **漏报** (High→Low) | 10倍 | 高危不能漏，极高惩罚 |
| **误报** (Low→High) | 1倍 | 浪费资源但可以接受 |

### 2. 期望损失 vs 实际损失

**风险评估器场景**: 使用**实际错误代价**（分类错误即产生代价）

**调度策略场景**: 使用**期望错误代价**（P(error) × C_error），因为LLM可能答对也可能答错

### 3. 代价权重对比

| 代价类型 | 场景一权重 | 场景二权重 | 说明 |
|:--------:|:----------:|:----------:|:----:|
| 延迟系数 | 0.01元/ms | 0.0005元/ms | 场景一更强调延迟 |
| 人工固定成本 | 200元 | 50元 | 场景一含转接成本 |

## 📐 公式汇总

### 场景一公式（分类器对比）

```
Error Cost = Σ [α_under × max(0, y-ŷ) + α_over × max(0, ŷ-y)] / N
           + C_human × I(ŷ ≠ y)

其中:
  α_under = 10, α_over = 1, C_human = 200
  y: 真实风险等级, ŷ: 预测风险等级

Latency Cost = latency_ms × 0.01

Total Expected Cost = Error Cost + Latency Cost
```

### 场景二公式（调度策略对比）

```
# 对每条请求
if 分配给LLM:
    Error Cost = P(error) × C_error
    Latency Cost = 16ms × 0.0005
    Human Cost = 0
else (分配给人工):
    Error Cost = 0
    Latency Cost = waiting_time × 1000 × 0.0005
    Human Cost = 50

Total Expected Cost = Σ(Error Cost + Latency Cost + Human Cost)
```

## 🎯 使用建议

1. **理解非对称性**: 漏报惩罚是误报的10倍，这是金融保守性的体现
2. **注意单位**: 场景一延迟系数是0.01，场景二是0.0005，不要混淆
3. **区分场景**: 场景一是分类器评估，场景二是调度系统评估，代价构成不同
4. **期望vs实际**: 调度场景使用期望损失（因为LLM有不确定性），分类场景使用实际损失
